# Embeddings & Vector Representations

Dense numerical vectors that encode semantic meaning. The backbone of modern search, recommendations, and LLM applications.

---

## What Is an Embedding?

A function that maps discrete, high-dimensional objects (words, images, users, products) into a continuous, lower-dimensional vector space where **similarity in vector space = similarity in meaning**.

```
"king" - "man" + "woman" ≈ "queen"    ← classic Word2Vec result
```

Nearby vectors are semantically related. Distance in vector space is meaningful.

---

## Types

### Text Embeddings
- **Word-level**: Word2Vec, GloVe — each word → vector.
- **Sentence/document-level**: BERT, Sentence-BERT, OpenAI Embeddings — full sentence or passage → single vector.
- Use case: semantic search, clustering, classification.

### Image Embeddings
- CNN or ViT backbone, final layer before classification head.
- Use case: visual similarity search, content moderation, product matching.

### Entity Embeddings (Collaborative Filtering)
- Users and items embedded jointly so user vectors cluster near items they like.
- Basis of most recommendation systems.
- Generated by matrix factorization or two-tower neural networks.

### Multi-Modal Embeddings
- Text and images in the same vector space (CLIP, ImageBind).
- Use case: search images with text queries, cross-modal retrieval.

---

## Generating Embeddings

```
[Input (text/image/entity)]
    → Embedding model (BERT, CLIP, two-tower)
    → Fixed-size dense vector (e.g., 768 dimensions)
    → Stored in vector DB
```

- Pre-trained models: use OpenAI, Cohere, or HuggingFace models via API or local inference.
- Fine-tuned models: train on your own data for domain-specific similarity (e.g., e-commerce products).

---

## Distance Metrics

| Metric | Formula | When to Use |
|---|---|---|
| **Cosine similarity** | angle between vectors | Text, normalized embeddings (most common) |
| **Dot product** | magnitude × angle | When magnitude encodes relevance (ranking) |
| **Euclidean (L2)** | straight-line distance | Image embeddings, when scale matters |

---

## Key Properties for System Design

- **Dimensionality**: higher = more expressive, slower to compare. Typical: 128–1536.
- **Normalization**: normalize to unit length if using cosine similarity (simplifies to dot product).
- **Freshness**: embeddings must be recomputed when items change significantly.
- **Storage**: N items × D dimensions × 4 bytes. 10M items × 768 dims = ~29GB.

---

## Embedding Pipelines

```
[Item added/updated] → [Embedding model] → [Vector DB upsert]
[Query arrives] → [Embed query] → [ANN search in Vector DB] → [Results]
```

Keep the same embedding model for indexing and querying — if they differ, distances are meaningless.

---

## Trade-offs

| Higher dimensionality | More expressive | Slower search, more memory |
| Fine-tuned model | Better domain relevance | Training cost, maintenance |
| Pre-trained model | Fast to deploy | May not capture domain-specific similarity |
| Approximate search (ANN) | Fast | Not exact — some relevant items missed |
| Exact search | Perfect recall | O(N) per query — too slow at scale |
