Start with the mental model first:

> Amazon Web Services S3 = **object storage**, not a traditional filesystem and not a block device.

You store **objects** (files + metadata) inside **buckets**. It’s designed for durability, scalability, and cheap storage.

For interviews, especially backend/devops/cloud roles, these are the areas you should actually know well:

---

# 1. Core Concepts

## Buckets

* Globally unique names
* Regional
* Containers for objects

Example:

```text
my-company-uploads
```

---

## Objects

Each object has:

* Key (path/name)
* Data
* Metadata
* Version ID (if versioning enabled)

Example:

```text
images/profile.png
```

S3 does not truly have folders. The `/` is just part of the key name.

That’s a very common interview question.

---

# 2. Storage Classes

Know the tradeoff:

* cost
* availability
* retrieval speed

Main ones:

| Class                      | Use Case            |
| -------------------------- | ------------------- |
| Standard                   | Frequently accessed |
| Standard-IA                | Infrequent access   |
| One Zone-IA                | Cheap, single AZ    |
| Glacier Instant Retrieval  | Archive but fast    |
| Glacier Flexible Retrieval | Cheap archive       |
| Glacier Deep Archive       | Very cold storage   |

Important idea:

> cheaper storage usually means slower retrieval.

---

# 3. Durability vs Availability

This gets asked constantly.

## Durability

S3 Standard:

```text
99.999999999% (11 nines)
```

Means your data is extremely unlikely to be lost.

---

## Availability

Usually:

```text
99.99%
```

Means service accessibility.

Interviewers love asking:

> “What’s the difference between durability and availability?”

Correct answer:

* Durability = data survives
* Availability = service reachable

---

# 4. Versioning

When enabled:

* deleting creates delete markers
* old versions remain
* protects against accidental overwrite/delete

Important for:

* rollback
* recovery
* ransomware mitigation

---

# 5. Lifecycle Policies

Automatic transitions/deletions.

Example:

* move to Glacier after 30 days
* delete after 365 days

This is huge for cost optimization questions.

---

# 6. Security

Very important interview area.

## IAM Policies

Identity-based permissions.

Example:

```json
{
  "Effect": "Allow",
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::my-bucket/*"
}
```

---

## Bucket Policies

Resource-based permissions attached to bucket.

Common interview question:

> IAM policy vs bucket policy?

Answer:

* IAM policy attached to users/roles
* Bucket policy attached to bucket resource

---

## Encryption

Know the main types:

| Type    | Managed By |
| ------- | ---------- |
| SSE-S3  | AWS        |
| SSE-KMS | AWS KMS    |
| SSE-C   | Customer   |

Most important in practice:

* SSE-S3
* SSE-KMS

KMS gives:

* auditability
* key rotation
* granular permissions

---

# 7. Pre-Signed URLs

Very common real-world pattern.

Backend generates temporary URL:

* upload directly to S3
* download securely

Benefits:

* avoids proxying files through backend
* scalable
* secure temporary access

This comes up constantly in system design interviews.

---

# 8. Multipart Uploads

Used for large files.

Benefits:

* parallel upload
* resume failed uploads
* better performance

Know:

> S3 recommends multipart uploads for large objects.

---

# 9. Consistency Model

This used to be tricky historically.

Modern S3:

* strong read-after-write consistency

Meaning:

* after PUT, reads immediately see latest object

Older interview material may mention eventual consistency. That is outdated.

---

# 10. Event Notifications

S3 can trigger:

* AWS Lambda
* SQS
* SNS

Example:

```text
image uploaded -> Lambda resizes image
```

Very common architecture question.

---

# 11. Replication

## Cross-Region Replication (CRR)

* replicate to another region

## Same-Region Replication (SRR)

* replicate within region

Used for:

* disaster recovery
* compliance
* latency optimization

---

# 12. Static Website Hosting

S3 can host:

* static HTML/CSS/JS sites

Usually combined with:

* Amazon CloudFront

Important distinction:

* static only
* no server-side code execution

---

# 13. Performance Concepts

Know these:

* virtually unlimited scale
* high throughput
* prefix partitioning mostly automatic now

Older advice about random prefixes is less important today.

# 16. Very Important Distinctions

## S3 vs EBS

| S3                   | EBS                |
| -------------------- | ------------------ |
| Object storage       | Block storage      |
| Massive scale        | Attached disk      |
| Access via API       | Mounted volume     |
| Not POSIX filesystem | Filesystem capable |

---

## S3 vs EFS

| S3             | EFS                     |
| -------------- | ----------------------- |
| Object storage | Network filesystem      |
| API-based      | NFS mount               |
| Cheap/scalable | Shared Linux filesystem |
